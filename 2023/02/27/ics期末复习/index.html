The JSR® instruction consists of three parts.

Opcode A Address evaluation bits
Bits [15:12] contain the opcode, 0100.

Bit [11] :addressing mode,

subroutine 保存为前者，调用者保存为后者.

Unit9 I/O

9.1 Privilege, Priority, and the Memory Address Space

9.1.1 Privilege and Priority

处理器状态寄存器（PSR）

  • PSR[15]：表示当前执行程序的模式，0为supervisor模式，1为用户模式。
  • PSR[10:8]：表示当前执行程序的优先级，从0到7，7为最高。
  • PSR[2:0]：表示条件码，N、Z、P分别表示最近一次修改寄存器的指令结果是否为负数、零或者正数。这些条件码用来判断分支指令是否跳转。

内存空间

内存地址空间 16 位，对应216个位置，每个位置包含一个字 (16位)。地址的 编号从 0 (x0000) 到 65535(xFFFF)。地址用于标识内存位置和内存映射 I/O 设备寄存器。

内存的某些区域是预留给特殊用途的:

  • 地址 x0000 到 x2FFF 包含特权内存（被称为系统空间），只有在 Supervisor 模式下 (PSR[15] = 0) 执行时才可以访问；

    x0000-x00FF Trap Vector Table; x0100-x01FF Interrupt Vector Table; 其中 x0100-x017F 是异常处理例程，x0180-x01FF 是中断服务程序。

  • 地址 x3000 到 xFDFF 包含用户模式 (PSR[15] = 1) 可用的内存；

  • 地址 xFE00 到 xFFFF 包含输入和输出设备寄存器和特殊的内部处理器寄存器，这些寄存器也只有在 Supervisor 模式下执行时才可以访问 (PSR[15] = 0)。为了控制对这些设备寄存器的访问，它们的地址也被认为是特权内存的一部分;

例如，PSR被分配地址xFFFC，处理器的主控制寄存器（MCR）被分配地址0xFFFE。第9.2.1.2节将讨论从内存地址空间分配地址的好处。从xFE00到xFFFF的地址集通常称为I/O页，因为大多数地址用于识别参与输入或输出功能的寄存器。访问这些寄存器需要管理员权限。

User stack

用户空间中的内存区域，可通过用户堆栈指针 (USP) 访问。当 PSR[15] = 1 时，堆栈指针 (R6) 是 USP。当运行在 Supervisor 模式 (PSR[15] = 0) 时， USP 被存储在 Saved_USP 中。

Supervisor stack

系统空间中的一个内存区域，可以通过管理器堆栈指针 (SSP) 访问。当 PSR[15] = 0 时，堆栈指针 (R6) 是 SSP。当处理器运行在 User 模式 (PSR[15] = 1) 时，SSP 被存储在 Saved_SSP 中。

超级用户栈供特权用户使用，和普通用户的栈空间分离。当中断发生时，用户模式由普通用户切换到超级用户。

超级用户栈指针存储在SAVE.SSP寄存器中。而普通用户栈指针存储在SAVE.USP中。值得注意的是，在LC-3中都是通过R6来访问栈指针的。

每个堆栈都有一个堆栈指针，即主管堆栈指针（SSP）和用户堆栈指针（USP），用于指示堆栈的顶部。由于一个程序在任何时候都只能在Supervisor模式或User模式下执行，所以在任何时候，两个堆栈中只有一个处于活动状态。寄存器6通常用作活动堆栈的堆栈指针（SP）。提供了两个寄存器，即保存的SSP和保存的USP，用于保存未使用的SP。当权限更改时，例如，从主管模式更改为用户模式时，SP存储在已保存的SSP中，SP从已保存的USP加载。

9.2 Input/Output

LC-3使用内存映射I/O。地址x000到xFDFF指的是实际的内存位置。地址xFE00至xFFFF保留用于输入/输出设备寄存器。表A.3列出了迄今已分配的LC-3设备寄存器的内存映射地址。

输入输出寄存器

即使是最简单的I/O设备通常也需要至少两个寄存器：一个用于保存设备和计算机之间传输的数据，另一个用于指示设备的状态信息。状态信息的一个示例是设备是否可用，或者它是否仍忙于处理最近的I/O任务。

TRAP

事实上，我们已经指出，TRAP指令的一个巨大好处是，它允许用户程序员不必知道本章前面讨论的I/O的可怕细节。此外，它还保护用户程序不受其他无能用户程序员的影响。

TRAP 指令周期的 EXECUTE 阶段做三件事:

  1. PSR, PC 入栈（此时 PC 已经自增，因此之后出栈的 PC 就是 TRAP 下一 条指令的位置）。如果是在用户模式下，R6 指向用户栈。在 PSR 和 PC 入 栈之前，需要执行 Saved_USP ← R6, R6 ← Saved_SSP。

  2. PSR[15] 被设置为 0 ，因为服务例程需要管理员权限才能执行。PSR[10:8] 保持不变，因为TRAP例程的优先级与请求它的程序的优先级相同。

  3. 8位的 trap vector 无符号扩展到 16 位，形成一个地址，对应于 TARP VECTOR TABLE 中的某个位置。该位置中的值（程序的起始位置）被加载 到 PC 中。

对应到状态机和数据通路里，完成如下操作：（参见 P713）

  • state15: TABLE ← x00, PC ← PC + 1, MDR ← PSR. TARP VECTOR TABLE 在内存中是 x0000-x00FF，因此 TABLE ← x00； PC + 1 是因为后面要保证后面出栈时的 PC 指向 TRAP 下一条指令；注 意此时的 PC 已经自增，这里又进行了一次 +1 看起来好像有问题，其实是因为后面 ，PC 会 -1；

  • state47: Vector ← IR[7 : 0], PSR[15] ← 0. Vector 存的是 trap vector，这里根据原来的 PSR[15] 确认下一状态，若为 0，直接进入 state37，若为 1，进入 state45；

  • state45: Saved_USP ← R6, R6 ← Saved_SSP. state45 表明原来是 User mode ，要向 Supervisor mode 转变，因此要先存 用户栈的栈顶指针，然后加载系统栈的栈顶指针.(书上 state45 有误，)

  • state37: MAR, SP ← SP − 1; state41: Write 这两个状态完成 PSR 入栈（根据 state15，此时 MDR 存的是 PSR ）

  • state43: MDR ← PC − 1; state46: MAR, SP ← SP − 1; state52: Write 这三个状态完成 PC 入栈。特别地 对于 INT，PC 已经在 state18 自增，但是由于中断，当前的指令也还没执 行，因此应该 PC-1； 对于 TRAP，前面 state15 已经 +1，所以这里 -1 也是对的，也就是前面 state15 的 PC + 1 实际上是为了和这里 PC - 1 统一）

  • state54、state53、state55 即加载 PC 为相应程序的起始地址，然后执行， 不赘述

RTI

首先检查 PSR[15] 若为 0 正常继续；若为 1，说明在用户模式下 RTI，则 发生异常，进行异常处理。

接着，PC, PSR 出栈；

然后再次检查 PSR[15]，若为 0，说明原来就是 Supervisor mode，不用切换，否则因此执行 Saved_SSP ← R6, R6 ← Saved_USP，完成 Supervisor mode 转变为 User mode；

具体到状态机里面其实和 TRAP 是镜像的，就不多赘述，出栈的PC就是原先程序要接着执行的正确的地址。

中断

进程的两个部分中断驱动I/O有两个部分：

  1. 使I/O设备能够中断处理器的机制。

  2. 处理中断请求的机制。

Part I: Causing the Interrupt to Occur

三个要求:

1.I/O设备必须需要服务。2.设备必须有权请求服务。3.设备请求必须比处理器当前所做的更紧急

大多数 I/O 设备中，有 Interrupt Enable(IE)，是设备状态寄存器的一部分。 IE 位为 [14] 位；

如果中断使能位 (位 [14]) 被清除，则与是否设置就绪位无关，I/O 设备将 不能中断处理器，因为它 (I/O 设备) 没有被授予中断处理器的权利。在这 种情况下，程序将不得不轮询 I/O 设备，以确定它是否准备好了；

如果设置了 [14] 位，则启用中断驱动的 I/O。在这种情况下，只要有人键 入一个键 (或者只要显示器完成对最后一个字符的处理)，就会设置位 [15]， 然后组合生成中断信号，进行处理； 为了停止处理器继续执行当前运行的程序并处理中断请求，必须INT信号。任何具有 [14] 和[15]位的设备都会发出中断请求信号。

中断请求信号被输入到优先级编码器，优先级编码器是一种组合逻辑结构，它从所有的请求中选择优先级最高的请求。如果该请求的PL高于当前执行程序的PL，则设置INT 信号。

发生中断时，令PSR, PC入栈，根据中断程序的优先级设置 PSR[10 : 8]，并且要 PSR[15] = 0（即工作在 Supervisor mode）;

同 TRAP 一样，如果原来是在用户模式下，R6指向用户栈。在 PSR 和 PC 入栈之前，需要执行，Saved_USP ← R6,R6 ← Saved_SSP；

中断和 TRAP 类似，Trap Vector Table 由 x0000 到 x00FF 的内存位置组 成，每个位置都包含一个 Trap 服务例程的起始地址；

而中断向量表由 x0100 到 x01FF 的内存位置组成，每个位置都包含一个中断服务程序的起始地址。处理器将 8−bit中断向量interrupt vector(INTV)展开，把形成的地址对应的中断向量表中位置的内容装入PC。

状态机来说： state49: TABLE ← x01, Vector ← INTV, PC ← PC + 1, MDR ← PSR, PSR[10 : 8] = Priority, PSR[15]− > 0 ,中断向量表在内存中是 x0000-x01FF，因此 Vector ← x01；注意没有 PC + 1，因为中断时该条指令还没有执行，返回时应该继续执行，所以这 里是自增后的 PC 入栈，后续 state43 又进行了一次 -1，则为当前 PC；

后续就和 TRAP 一样，根据原来的 PSR[15] 确认下一状态，若为 0，进入 state37，若为 1，进入 state45，之后的都一样，不赘述。 中断返回，用的也是 RTI。

中断过程

中断分为四个过程：

  • 保存现场，即保存中断发生时的系统状态，比如PC和PSR寄存器

  • 计算中断向量：通过中断向量寻址找到中断服务程序起始位置

  • 执行中断服务程序

  • 从栈中还原中断发生前的现场

具体如下:

1.中断过程的PSR保存在TEMP中。

2.处理器将特权模式设置为主管模式（PSR[15]=0）。

3.处理器将优先级设置为PL4，即中断设备的优先级（PSR[10:8]＝100）。

4.如果中断的进程处于用户模式，则R6保存在保存的USP中，R6加载有监管堆栈指针（SSP）。

5.TEMP和中断进程的PC被推到监管器堆栈上。

6.键盘提供其八位中断向量，在本例中为x80。

7.处理器将该向量扩展到x0180，即中断向量表中对应的16位地址。

8.PC加载有存储器位置x0180的内容，即键盘中断服务例程中第一条指令的地址。

中断返回

和RET不同，使用RTI指令以从中断或者trap中返回。

和RET不同的点在于，RTI主要做了两次退栈的操作，即弹出PC和PSR。

Problem

  • 这个问题的原因是轮询和中断服务之间的优先级冲突。当键盘中断发生时，它会打断轮询循环，并执行键盘中断服务例程。这个例程会向DDR写入一个字符，但是没有更新轮询循环的状态寄存器。

  • 当键盘中断服务例程完成后，轮询循环会恢复执行，并认为DDR仍然准备好接收数据，所以它会执行STI指令。但是由于DDR还没有完成上一次的写入操作，所以STI指令会失败，并丢失一个字符。

  • 这个问题的解决方法是在键盘中断服务例程中保存和恢复状态寄存器的值。(当然也可以在轮询时禁用中断但是不推荐)这样可以保证轮询循环在恢复执行时能够正确地检测DDR的状态，并等待它准备好再写入数据。

另一种方式是在轮询中不断启用又禁用中断，但是也有一些缺点。比如：

  • 在轮询中不断启用又禁用中断会增加CPU的开销，因为每次启用或禁用中断都需要修改PSR寄存器。

  • 在轮询中不断启用又禁用中断会降低系统的响应性，因为如果在禁用中断的时候有一个高优先级的中断发生，那么CPU就无法及时处理它，而必须等到轮询循环再次启用中断。

LC-3寄存器

  • GPR

GPR是General Purpose Register的缩写，意思是通用寄存器。LC-3有8个通用寄存器，分别是R0到R7，每个寄存器都是16位的。你可以用这些寄存器来存储数据或者地址，比如在算术或者逻辑运算中。

  • PC：程序计数器，用来存储下一条指令的地址。

  • IR：指令寄存器，用来存储当前执行的指令。

  • PSR：程序状态寄存器，用来存储一些状态信息，比如条件码和优先级。

    • PSR[15]：表示当前执行程序的模式，0为系统模式，1为用户模式]。
    • PSR[10:8]：表示当前执行程序的优先级，从0到7，7为最高。
    • PSR[2:0]：表示条件码，N、Z、P分别表示最近一次修改寄存器的指令结果是否为负数、零或者正数。这些条件码用来判断分支指令是否跳转。

  • MPR：内存保护寄存器，用来检查内存访问是否合法。

    它的作用是检查内存访问是否合法，防止用户程序执行系统代码或者访问设备。MPR有16位，每一位对应一段内存区域，如果该位为1，表示该区域可以在用户模式下执行；如果为0，表示只能在系统模式下执行。如果用户程序试图执行不允许的区域，会触发异常并停止执行.

    比如说，假设MPR的第0位为0，表示内存地址x0000到x0FFF的区域只能在系统模式下执行。如果用户程序想要执行这个区域的代码，比如TRAP指令，就会被MPR拦截并抛出异常。这样就保证了用户程序不能随意调用系统服务或者修改系统数据。

  • MAR：内存地址寄存器，用来暂时保存要访问的内存地址。

  • MDR：内存数据寄存器，用来暂时保存要读取或者写入的内存数据。

以FETCH指令为例

FETCH阶段从存储器访问开始，以在PC指定的地址读取指令。注意，在编号为18的状态中，MAR被加载有包含在PC中的地址，并且在当前指令完成其指令周期之后，PC被递增以准备下一个LC-3指令的FETCH。

如果MAR的内容指定了特权内存，并且PSR[15]=1(表示用户模式)，则不允许访问指令。这将违反访问控制，因此设置了ACV。最后，如果不存在中断请求（INT=0），则流程转到编号为33的状态。我们将在C.7节中描述INT=1时的控制流程，即外部设备是否请求中断。(我们也将讨论如果第C.7节中存在ACV违规行为会发生什么。)

现在，回到在状态18的活动已经完成的时钟周期之后发生的事情。正如我们所说的，如果没有外部设备请求中断，则流程转到状态33（即100001）。如果处理器在用户模式下试图访问特权存储器，则控制从状态33转到状态60，如果允许存储器访问，即如果没有ACV违规，则控制转到状态28。

在状态28中，由于MAR包含要处理的指令的地址，因此该指令从存储器中读取并加载到MDR中。由于该存储器访问可能需要多个周期，因此该状态继续执行，直到来自Memory-ready signal（R）被断言，表明存储器访问已完成。因此，MDR包含MAR指定的存储器位置的有效内容。然后状态机进入状态30，在状态30中，指令被加载到指令寄存器（IR）中，完成指令周期的FETCH阶段。

状态机然后移动到状态32，在那里进行DECODE。注意，有13个弧从状态32发出，每个弧对应于LC-3指令的位[15:12]。这些操作码位将状态机引导到16条路径之一，以执行刚刚获取的特定指令的指令周期。注意，从每个指令周期的最后一个状态（即完成该LC-3指令处理的状态）开始的弧将我们带到状态18（开始下一个LC-3指令的指令周期）。

PCMUX还包括两个控制信号，并指定在给定时钟周期内需要MUX的哪个输入。

LDPC是一个单比特控制信号，并且是0（否）或1（是），这取决于在给定时钟周期内是否要加载PC。

假设读取一个值需要五个周期的内存。也就是说，一旦MAR包含要读取的地址，并且微指令断言read，则需要五个周期才能将存储器中指定位置的内容加载到MDR中。（注意，微指令通过两个控制信号断言read：MIO.EN/YES和R.W/RD；）

特别的,LD.reg指令在数据通路中需要以下控制信号:

  • LD = 1，表示要执行LD指令

  • PC = 1，表示要将PC的值放到总线上

  • MAR = 1，表示要将总线上的值存入MAR寄存器

  • MIO.EN = 1，表示要从内存中读取数据

  • R.W = 1，表示要将MDR寄存器的值写入目标寄存器

  • DRMUX = reg[11:9]，表示要选择LD指令中指定的目标寄存器

所以LD.reg在数据通路中为0/1的意思是LD.reg是一个控制信号，它为1时表示执行LD指令，为0时表示不执行。
